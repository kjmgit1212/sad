Running the WebServices API Examples

Overview

The following are working examples that use the WebServices API. This API allows access to various operations within the provisioning system.

When working with the WebServices API, it is highly recommended that any operations to be performed are first tested manually via the standard IBM Security Identity Manager UI.

The source code for the examples can be found in the extensions/{RELEASE_VERSION}/examples/ws/src directory. The scripts to run the examples can be found in the extensions/{RELEASE_VERSION}/examples/ws/bin/win directory for Windows and extensions/{RELEASE_VERSION}/examples/ws/bin/unix directory for UNIX.

Building the Examples

To build the examples follow the instructions given in Building the Examples.

Installing the Examples on WebSphere

There is nothing additional that needs to be done if the examples are executed from their default location (IBM Security Identity Manager installation area). The examples can also be configured to run on a separate system. The instructions to setup the examples remotely are out of scope of this document. Refer to the instructions given in the API Packaging Guide to configure the IBM Security Identity Manager WebServices APIs.

Using ws_example_config.properties file

Copy the ws_example_config.properties file from the extensions/{RELEASE_VERSION}/examples directory to the following locations on Windows and Unix systems:

On Windows:
extensions/{RELEASE_VERSION}/examples/ws/bin/win


On Unix:
extensions/{RELEASE_VERSION}/examples/ws/bin/unix

Modify the appropriate file and specify the correct host and port values. This properties file provides the host and port information to the examples.

Running the Examples

Build the Examples before running them. Make sure the IBM Security Identity Manager server is running.

Logging in to IBM Security Identity Manager

The WSSessionServiceClient.java source file demonstrates how to log in IBM Security Identity Manager.

Execute the accompanying wsSessionService.sh or wsSessionService.cmd file by using the following command syntax:

Windows usage:

usage: wsSessionService -operationName?login -[argument]?[value]

-operationName        Name of the operation from the WSSessionServiceClient which is to be executed.
-principal            User ID for logging in to TIM for example "itim manager"
-credential           Password for logging in to TIM

Example: wsSessionService -operationName?login -principal?"itim manager" -credential?"secret"

The command line syntax is a list of name/value pairs in the format –[argument] ? [value]. All the parameters are required for executing the login operation.

The WSSessionServiceClient.java source file demonstrates how to login in to IBM Security Identity Manager using the challenge responses and new password.

Execute the accompanying wsSessionService.sh or wsSessionService.cmd file by using the following command syntax and argument list:

Windows Usage:

usage: wsSessionService -operationName?lostPasswordLoginDirectEntry -principal?<username>
-newCredential?<newpassword of user> -locale?<Language code> -answers?<answer(s) of a challenge question(s)>

-principal        User name for logging in to IBM Security Identity Manager (For example, "itim manager")
-newCredential     New password of the user.

In the above example, we had two challenge questions set namely:
1. Which is your favorite color?
2. What is your school name?

Modify Person

The WSPersonServiceClient.java source file demonstrates how to modify a person in IBM Security Identity Manager.

Execute the accompanying wsPersonService.sh or wsPersonService.cmd file by using the following command syntax and argument list:

Windows Usage:

usage: wsPersonService -operationName?modifyPerson -principal?<username> -credential?<password>
-filter?<LDAP Filter> -attributeName?<attributeValue>:<modOperationType>

-principal         User name for logging in to IBM Security Identity Manager (For example, "itim manager")
-credential        Password of the user to log in to IBM Security Identity Manager.
-filter            LDAP filter for searching the person.
-attributeName     Valid attribute names and their values which are to be modified. For multivalued attributes please specify the same attribute name for each value. The modOperationType argument should be used with values REPLACE, ADD OR REMOVE. This informs the client if the attribute has to be replaced, added (in case of multivalued attributes) OR removed respectively. In case of multivalued attributes you can do a selective remove.


Example: wsPersonService -operationName?modifyPerson -principal?"itim manager" -credential?"secret" -filter?"(&(cn=Judith Hill)(sn=Hill))" -mail?"jhill@ibm.com:ADD" -mail?"jhill@personalEmail.com:ADD -employeeNumber?"09A4892":REMOVE -alias?"judithH":REPLACE

In the above example we are modifying the person Judith Hill by adding a couple of emails to the mail attribute, removing the employeeNumber attribute and replacing all aliases with judithH.

-principal          User name for logging in to IBM Security Identity Manager. (For example, "itim                     manager")
