Overview


You can use the self-care application to enable users to manage their user IDs and passwords. Enabling users to perform these types of self-management tasks reduces the number of help desk calls.

NOTE: This self-care application is different than the Self Service application that comes with IBM Security Identity Manager. This example demonstrates how to write your own self-care web application by using IBM Security Identity Manager public APIs. This example is for the demonstration purpose only. The self-care example application supports the following operations:

  • Changing your password
  • Changing your personal information
  • Changing your challenge response answers
  • Registering yourself as a new user
  • Resetting your password by answering the challenge response questions

Building the Self-Care Application


Use the build scripts in the examples directory to create itim_expi.war file. See Building the Examples for more details about building examples.
  • Make sure that the IBM Websphere Application Server is installed on the client where you want to build the self care application.
  • Set the WAS_HOME environment variable to point to the WebSphere Application Server installation directory.
  • To build only the self-care application, specify self_care as the target when you build the examples. For example, issuing the "build self_care" command on Windows operating systems, or the "./build.sh self_care" command on UNIX operating systems, creates the itim_expi.war file. The administrator installs this war file onto WebSphere Application Server by following these instructions. If the administrator makes any changes to the source of the self-care application, the administrator must recreate itim_expi.war file.


Configuring Websphere Application Server before installing the Self-Care Application


  • Make sure that administrative security is enabled for the profile on which the Self Care Application is to be installed.
  • Create a folder named classes (if it does not exist) in $WAS_HOME/profiles/profile_name.
  • Copy the itim_common.jar, com.ibm.cv.kmip.ext.jar, itim_util.jar, and jlog.jar files from extensions/lib to the $WAS_HOME/profiles/profile_name/classes folder.
  • Copy the isimva_user_registry.jar file from extensions/{RELEASE_VERSION}/user registry to $WAS_HOME/profiles/profile_name/classes folder.
  • Edit the isimva_registry.properties file in extensions/{RELEASE_VERSION}/user registry for the Self Care Application.
    • From the IBM Security Identity Manager virtual appliance console, click Configure > Update Property.
    • Select the enRole.properties file from the All properties > Identity server properties file tab.
    • Search for the value of following properties and update the corresponding property in the isimva_registry.properties file.
      • enrole.ldapserver.root
      • enrole.defaulttenant.id
      • enrole.appServer.realm
      • enrole.encryption.passwordDigest
    • Select the enRoleAuthentication.properties file from the All properties tab.
    • Search for the value of following properties and update the corresponding property in the isimva_registry.properties file.
      • enrole.authentication.requiredCredentials
      • enrole.authentication.provider.simple
    • For the last set of properties, contact your directory server administrator.
      • java.naming.provider.url = LDAP URL in the format ldap://IP address of LDAP server:port
      • java.naming.security.principal = LDAP principal Distinguished Name
      • java.naming.security.credentials = LDAP principal Distinguished Name password
      • java.naming.security.protocol = Security protocol to be used with LDAP. For example, SSL. Remove this property if no security protocol is defined for the directory server.
  • Copy the isimva_registry.properties in the WAS_HOME/profiles/profile_name /properties folder on the WebSphere Application Server.
  • Restart the WebSphere Application Server.
  • Log on to the WebSphere Administrative console and click Security > Security Domains > Copy Global Security.
  • Enter the name of domain ISIMSecurityDomain, click OK and save the changes to the master configuration.
  • Configure the security domain.
    • Go to Security > Security Domain > ISIMSecurityDomain. Specify server1 as the scope of the domain. Click OK and save the changes to the master configuration.
    • Go to Security > Security Domain > ISIMSecurityDomain > User Realm. Select Standalone custom registry.
    • Click Configure and enter the realm name as itimCustomRealm and custom registry class name as com.ibm.isimva.registry.IsimVaUserRegistry. Select Ignore case for authorization.
    • Click OK and save the changes to master configuration.
  • Configure SSL appropriately between the IBM Security Identity Manager virtual appliance and the WebSphere Application Server. For example:
    • Log on to the administrative console of the WebSphere Application Server.
    • Expand Security and click SSL certificate and key management.
    • Under Related Items, click Key stores and certificates and click the NodeDefaultTrustStore key store.
    • Under Additional Properties, click Signer certificates and click Retrieve From Port.
    • In the Host field, enter the IP address of Application Interface in the IBM Security Identity Manager virtual appliance.
    • In the Port field, enter 9072. Set Alias as the IP address of Application Interface suffixed by _cert.
    • Click Retrieve Signer Information.
    • Verify that the certificate information is for a certificate that you can trust.
    • Click Apply and Save.
  • Ensure that you export and import the LTPA keys for the encryption and decryption of the identity tokens.
    • To export the LTPA key from the IBM Security Identity Manager virtual appliance, do these steps.
      • On IBM Security Identity Manager virtual appliance console, click Configure > Single Sign-On Configuration.
      • Select LTPA Keys tab.
      • Specify a password in the Password and Confirm password fields.
      • Click Export LTPA keys.
    • Import the LTPA key on the WebSphere Application Server.
      • Go to Security > Global Security > LTPA.
      • Specify the password that was used in exporting the ltpa key in the Password and Confirm Password fields.
      • Specify the path of the LTPA key file downloaded from the virtual appliance in the Fully qualified key file name field.
      • Click Import keys.
      • Save the changes to the master configuration.
  • Restart the WebSphere Application Server server.


Installing the Self-Care Application on WebSphere


Before you install the self-care application, be familiar with the requirements for installing it and the methods of installation.

You must install the self-care application on a system that has WebSphere Application Server already installed. You must also have installed the WebSphere Application Server patches that are specified in the IBM Security Identity Manager Release Notes. Use the installation instructions in the Release Notes to install the patches.

Follow these steps to install the self-care application on a WebSphere Application Server:

  • Build the self-care application to create the itim_expi.war file.
  • Log on to the WebSphere administrative console.
  • Under Applications -> New Application click New Enterprise Application.
  • For Path to the new application, select Local file system and set Full path to the location of the itim_expi.war file.
  • For installation method, select Detailed - Show all installation options and parameters.
  • For Map context roots for Web modules, specify /itim_expi.
  • For Map security roles to users or groups step, map SELFCARE_CLIENT role to the special subject, All Authenticated in Trusted Realms.
  • Repeatedly click Next until you arrive at the Summary window.
  • At the Summary window, click Finish.
  • Click Save to save your changes directly to the master configuration.
  • Under Applications -> Appliation Types -> WebSphere enterprise applications, select itim_expi_war.
  • Under Detailed Properties -> Class loading and update detection, select Classes loaded with local class loader first (parent last) for Class loader order and select Single class loader for application for WAR class loader policy.
  • Click OK to apply the changes.
  • Click Save to save the changes to the master configuration.
  • Do not start the self-care application until after you have followed the remaining steps, including Changes if WebSphere Java 2 Security is Enabled, Configuring the Self-Care Application, and Configuring IBM Security Identity Manager.
  • Copy the itim_expi.properties file from the extensions/{RELEASE_VERSION}/examples/self_care directory to the $WAS_HOME/properties directory.

Changes if WebSphere Java 2 Security is Enabled


If Java 2 security has been enabled for the WebSphere Application Server where the self-care application has been installed, follow this procedure:

  • Edit the was.policy file in the $WAS_HOME/profiles/$PROFILE_NAME/config/cells/$CELL_NAME/applications/itim_expi_war.ear/deployments/itim_expi_war/META-INF directory (The $WAS_HOME is the directory where WebSphere Application Server is installed. The $PROFILE_NAME is the name of the WebSphere Application Server profile in which the self-care application has been installed. The $CELL_NAME is the name of the WebSphere Application Server cell in which the self-care application has been installed).
  • Modify the application permissions in the was.policy file as follows:
    grant codeBase "file:${application}" {
    permission java.security.AllPermission;
    };
  • Restart the WebSphere Application Server on the system where the self-care application has been installed.

Configuring the Self-Care Application

Edit the itim_expi.properties file in the $WAS_HOME/properties directory (where $WAS_HOME is the directory where WebSphere Application Server is installed on the system where the self-care application is installed), and set the following key properties:

Tenant and Tenant DN setup:
tenantid=<your tenant ID>
tenantdn=<your tenant DN>

Default organization (root in Identity Manager):
default.org=root organization in IBM Security Identity Manager
platform.url=corbaloc:iiop://host name of IBM Security Identity Manager server:port
 (URL where Identity Manager is installed)

platform.principal=IBM Security Identity Manager administrator
platform.credentials=IBM Security Identity Manager password

You can determine the values for these properties by looking at the corresponding values in the enRole.properties file. The file can be read from IBM Security Identity Manager virtual appliance console. Click Configure > Update Property. Select enRole.properties file from the All properties > Identity server properties file tab. The properties and corresponding values are described in the following table.

Property in itim_expi.properties file Corresponding value in the enRole.properties file
tenantid Use the value for enrole.defaulttenant.id.
tenantdn Use the value "ou=tenantid" combined with the value of enrole.ldapserver.root. For example, if the value of tenantid were "myco", and the value of enrole.ldapserver.root were "dc=com", the property could be set using "tenantdn=ou=myco,dc=com".
default.org Use "ou=tenantid". For example, if the value of tenantid were "myco", the property could be set using "default.org=ou=myco".
platform.url Use the value for enrole.appServer.url. Note: If the value of enrole.appServer.url contains "localhost", substitute the host name or IP address of the system where IBM Security Identity Manager is installed.
platform.principal Use the name of IBM Security Identity Manager administrator.
platform.credentials Use the password of IBM Security Identity Manager administrator.


Following are example values for these key properties in the itim_expi.properties

file:
#------------------------------------------------------
# Organizational information
#------------------------------------------------------
tenantid=myco
tenantdn=ou=myco,dc=com
default.org=ou=myco

# Application Server
platform.url=corbaloc:iiop://itimserver.myco.com:9067
platform.principal=itim manager
platform.credentials=admin

Download tmsMessages_en.properties. The file can be downloaded from IBM Security Identity Manager virtual appliance console. Click Configure > Custom File Management. Select nls folder from All files > directories. Search for tmsMessages_en.properties. Click on Dowload button. Copy this file to $WAS_HOME/properties directory

Configuring IBM Security Identity Manager


Log on to the Identity Manager administrative console and follow these steps to configure Identity Manager:
  • Create a Location in the IBM Security Identity Manager organization structure for the self-registered users. By default, the self-care application uses the Location selfregisterhere. You can specify any name but you must update the self-care application to recognize the name.

    If you create a non-default Location name, edit the itim_expi.properties file to update the self-care application. This file is located in $WAS_HOME/properties directory (where $WAS_HOME is the directory where WebSphere Application Server is installed on the system where the self-care application is installed), and set the following property to the name of the Location: 
    orgContainer.selfregister.location.org=selfregisterhere

  • Enable forgotten password authentication using administrator-defined questions. For more information see IBM Security Identity Manager Information Center.

Starting the Self-Care Application


Start the self-care application running on the WebSphere Application Server:
  • Log on to the WebSphere administrative console running on the system where the self-care application is installed (for example, http://localhost:9060/ibm/console).
  • Navigate to Applications > Enterprise Applications.
  • If itim_expi_war is already stopped then go to the next step. Otherwise check the itim_expi_war checkbox and click Stop.
  • Check the itim_expi_war checkbox and click Start.

Testing the Self-Care Application


In a Web browser window enter the URL http://hostname:9080/itim_expi or if SSL has been enabled, https://hostname:9443/itim_expi (Where hostname is the name of the host where the self-care application is installed). The self-care application Sign On page is displayed. Try various tasks. If you encounter problems, verify that IBM Security Identity Manager is configured to enable the user to perform the task (that is, make sure the same operation can be performed by the same user using the standard Security Identity Manager user interface).
